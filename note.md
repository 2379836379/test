# note

## 可靠性机制
1. 每个顶点有两个位图   

sending bitmap：记录这个顶点发出去之后，哪些邻居已经确认收到了  
receiving bitmap：记录这个顶点需要收的邻居特征里，哪些已经到达了  
这里“一个 bit 对应一个邻居”。如果某个顶点有 4 个邻居，那这两个位图都可以理解成 4 位状态表。

2. 收到聚合结果后怎么处理

如果主机收到的是“交换机已经聚合完成的结果”
那说明这个顶点需要的邻居特征已经齐了
这时把它的 receiving bitmap 直接置成全 1
然后这个 worker 再给这些邻居发 ACK
意思是：“我这个顶点依赖的邻居数据已经完整到手了，你们之前发给我的这批数据可以算成功了。”
当某个顶点的 sending bitmap 也变成全 1 时，就表示：这个顶点发出去的数据，所有相关邻居都确认完成了，发送过程彻底结束。

3. 为什么要区分‘聚合结果’和‘未聚合特征’
论文说，主机收到一个数据包时，先要判断：

这是交换机返回的“完整聚合结果”
还是只是某个邻居原始发来的“未聚合特征”
这很重要，因为两者后续处理不同：

完整聚合结果：可以直接用于当前顶点计算，并把接收位图置满
未聚合特征：不能直接当最终结果，需要先缓存起来

4. 本地 cache 是干什么的
如果收到的是“未聚合特征”，主机会把它缓存到本地。

原因是：
同一台 worker 上，可能有多个 boundary vertices 都需要同一个远端顶点的特征。

如果已经有一个顶点把这份远端特征收到了，那别的本地顶点就没必要再去远端重新拉一次。
直接从本地 cache 共享就行。

5. 会收到未聚合特征的典型情况是：

超时恢复 / FETCH
某个 host 等太久，还没等到当前 block 的聚合结果
它就去别的 worker 主动拉缺失特征
这些被拉回来的包会被标成 bypass
交换机看到这种包后，不再参与聚合，而是直接转发给 host
于是 host 收到的就是“未聚合特征”
重传 / RESEND
为了避免重复累加，重传包不能再进原来的聚合器
交换机会释放对应聚合器，并把这个重传包直接转发给目标 host
这时 host 收到的也是“未聚合原始特征”，要在主机侧自己处理

6. 超时恢复机制。

恢复流程

某个 worker 等超时了
它发现当前 block 迟迟没有收到需要的特征。

它主动去别的 worker 拉缺的特征
也就是不再等交换机聚合完成，而是自己发起请求，把缺失的原始特征直接要回来。

被拉回来的这些包到达交换机后，不再进聚合器
论文里把这类包标成 bypass。

bypass 的意思
不是再参与交换机内聚合，而是直接转发给目标 host。

后续聚合改在 host 侧完成
因为交换机这条聚合路径已经被判定“卡住了”，所以剩下的补齐和求和工作由主机自己做。

7. “释放对应 aggregator”是什么意思

原来交换机里有一个槽位正在等这一批包凑齐
现在 host 已经决定绕过交换机，自己拉原始特征来补
那这个交换机里的 aggregator 就没必要继续占着了
所以把它清掉、释放掉，避免一直卡住资源
为什么要这么做
因为交换机聚合很快，但前提是包能按时到齐。
一旦有丢包，继续死等会让整个 block 卡住。
超时后改成 host 主动拉取，相当于从“交换机内聚合快路径”切到“主机侧恢复慢路径”，保证最终还能完成。

一句话总结
这段的意思就是：

没丢包时：走交换机聚合
丢包超时时：host 主动拉缺失特征
拉回来的特征打上 bypass
交换机直接转发，不再聚合
最终由 host 自己补齐并完成聚合

8. 拥塞控制，本质上是：

平时主机按既定调度顺序持续发送顶点
如果网络出现拥塞，交换机会在包上打 ECN 标记
发送端看到 ECN 后立刻降速
如果之后一段时间都没有新的拥塞信号，就再慢慢提速
可以拆成 3 步理解。

按调度顺序持续发送
这里不是“发一个等一个”，而是 worker 按预先安排好的顶点顺序不断往外发。
这样能保持交换机聚合流水线一直有数据，提高吞吐。

用 ECN 判断是否拥塞
ECN 是显式拥塞通知。

意思是：

交换机发现自己队列太长、快堵了
但它不一定直接丢包
而是在经过的包头上打一个拥塞标记 ECN
发送端收到这个被标记的包，就知道：
“网络现在拥塞了，我发得太快了。”

拥塞时乘性减小，不拥塞时加性增大

具体就是：

收到 ECN 标记时，发送速率减半
哪个 worker 收到了 ECN 标记，哪个 worker 就降低自己的发送速率。
不是严格要求发送端降速

这是快速降速
目的：立刻缓解交换机队列压力
如果网络不再拥塞，发送速率线性增加

不是一下恢复到原来速度
而是慢慢一点点加快
这样更稳，不容易刚恢复就再次打满网络
这就是典型的：

乘性减小：拥塞了就砍一半
加性增大：不拥塞了就慢慢加
为什么这样设计
因为在网聚合对交换机很敏感：

发太快：交换机队列堆积，可能丢包，聚合卡住
发太慢：交换机吃不满，吞吐低
所以它需要一个动态调速机制：
既尽量跑满网络，又别把交换机压爆。

一句话总结
这段的意思是：

SwitchGNN 发送端会持续发顶点；交换机用 ECN 告诉发送端“现在堵了”；发送端一旦看到 ECN 就把速率减半；如果后续没再堵，就再慢慢把速率加回来。

9. 现在每个 MTP 包的有效载荷最多就是 128 字节。
依据是：
PAYLOAD_LEN 定义为 128，见 common.h (line 22)
send_msg() 里如果 plen > PAYLOAD_LEN 会直接返回失败，见 lab_host.c (line 600)
router 聚合器也是按这个固定长度累加的，见 common.h (line 220)
所以更准确地说是：
每个包的数据区最多 128B
不包含以太网头、IP 头、MTP 头
当前还没有实现多包分片/重组
