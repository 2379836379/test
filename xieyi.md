# SwitchGNN 论文中的网络协议说明

本文对应的“网络协议”并不是一个独立命名的通用传输协议，而是论文 **SwitchGNN** 为分布式 GNN 训练设计的一套 **主机-交换机协同通信机制**。它建立在 IP + DPDK + P4 可编程交换机之上，核心目标是在图传播阶段把传统主机侧的：

- 一对多特征复制（multicast）
- 多对一邻居聚合（aggregation）

尽量搬到交换机中完成，从而减少网络中的冗余流量。

## 1. 协议参与方

- `Worker/Host`：保存本地子图与顶点特征，按预处理后的顺序发送边界顶点特征。
- `P4 Switch`：负责入网复制、交换机内聚合、块同步、ECN 标记。
- `Aggregator`：交换机内的状态槽位。每个槽位对应一个目标顶点的聚合上下文。

协议的前提是：离线预处理已经完成。

- 使用 `GAMR`（Graph-Aware Multicast Reordering）决定顶点发送顺序。
- 使用多级图划分把存在依赖的边界顶点切成若干 `block`，保证单个 block 能放进交换机聚合资源。

## 2. 数据包头格式

论文在实现部分给出了 SwitchGNN 自定义头字段，插在 IP 头之后：

- `Src_id`：32 bit，源顶点 ID
- `Dst_id`：32 bit，目标顶点 ID
- `Block_id`：16 bit，当前传输批次/块编号
- `Count`：16 bit，当前顶点涉及的边界邻居数量
- `Is_ACK`：1 bit，ACK 包标记
- `Is_Fetch`：1 bit，拉取/补取包标记
- `ECN`：1 bit，显式拥塞通知标记
- `Resend`：1 bit，重传包标记

这些字段共同承担“协议状态编码”的作用，而不是依赖 TCP 连接语义。

## 3. 基本通信流程

### 3.1 预处理阶段

主机侧提前得到两类元数据：

- 每个边界顶点属于哪个 `block`
- 每个 block 内顶点的发送顺序

交换机侧提前安装映射表：

- `vertex ID -> aggregator index/集合`

也就是：某个源顶点到达交换机后，应该被复制到哪些聚合器，由表项直接决定。

### 3.2 block 内发送

每个 worker 按如下规则发送：

1. 只发送当前 `block_id` 内的本地边界顶点。
2. 发送顺序遵循 GAMR 排好的顶点序列。
3. 当前 block 未完成前，不能发送下一 block。

这样做的原因是：

- 减少随机发送导致的交换机等待和队列堆积。
- 限制同时活跃的依赖集合，避免 aggregator overflow。

### 3.3 交换机内 multicast + aggregation

一个顶点特征包到达交换机后：

1. 交换机根据 `Src_id` 查表，得到它需要参与的目标聚合器集合。
2. 将该顶点特征复制到多个目标聚合器中。
3. 每个聚合器持续累加来自不同邻居的特征。
4. 当聚合器收齐所需输入后，生成一个聚合结果包并发回对应 worker。
5. 对应聚合器槽位释放，供后续顶点复用。

论文实现里并不是用 bitmap 判断“哪些邻居已经到齐”，而是依据 `Count` 相关的聚合计数完成判断。

### 3.4 block 完成同步

当某个 worker 收到当前 block 的全部聚合结果后：

1. 向交换机发送 `block_id + host_id` 的完成通知。
2. 交换机记录当前 block 已完成的 host 数量。
3. 当所有 host 都完成后，交换机向所有 worker 组播 block 完成信号。
4. 所有 worker 开始发送下一个 block。

这一步是协议中的一个关键屏障（barrier），保证跨主机状态一致。

## 4. 可靠性机制

由于一个包在交换机里会被“一变多”，多个包又会被“多变一”，传统端到端语义不再足够，所以论文单独设计了可靠性协议。

### 4.1 双 bitmap 跟踪

对每个顶点，主机维护两类位图：

- `sending bitmap`：记录该顶点发送后的 ACK 状态
- `receiving bitmap`：记录该顶点所需邻居特征的接收状态

处理逻辑：

- 当 worker 收到某顶点的完整聚合结果时，对应 `receiving bitmap` 置全 1。
- 然后该 worker 向该顶点的邻居发送 ACK。
- 当 `sending bitmap` 也变成全 1，说明该顶点的这轮传输完成。

### 4.2 非聚合结果的本地缓存

如果 worker 收到的不是交换机返回的完整聚合结果，而是未在交换机中完成聚合的特征：

- 先缓存到本地。
- 若本机其他边界顶点后续也需要该特征，可直接复用缓存，避免再次跨机拉取。

这说明协议允许“部分流量回退到主机侧聚合”，并通过缓存降低额外开销。

### 4.3 超时与补取

当网络拥塞或丢包导致某 worker 在超时时间内还没收到当前 block 所需特征时：

1. worker 直接向其他 worker 拉取缺失特征。
2. 这些拉取包被标记为 `Is_Fetch`/bypass。
3. 一旦这类包到达交换机，对应聚合器被释放。
4. 被拉取到的特征不再参加交换机内聚合，而是直接转发给主机做最终聚合。

因此，协议不是“必须完全依赖交换机聚合”，而是带有故障退化路径。

## 5. 重传与重复累加处理

若发生重传：

- 重传包的 `Resend = 1`
- 交换机看到该标记后，不再把它继续计入原聚合器
- 而是释放相应聚合器，并把该包直接转发到目标 worker 侧聚合

这样可以避免因为重复包再次参与交换机聚合而造成 **duplicate accumulation**。

## 6. 拥塞控制机制

SwitchGNN 的速率控制类似 DCQCN：

- 发送方持续按照调度顺序发送顶点
- 交换机队列超过阈值时，在包头中打 `ECN`
- 发送方收到 ECN 后将发送速率减半
- 若网络未拥塞，则发送速率加性增加

因此它属于：

- 交换机负责拥塞显式反馈
- 主机负责速率调整

的协同拥塞控制方案。

## 7. 用协议术语总结 SwitchGNN

如果把论文内容整理成协议视角，可以概括为：

- `传输单位`：顶点特征包
- `通信域`：跨 worker / 跨 rack 的边界顶点通信
- `调度单位`：`block`
- `交换机状态`：aggregator 槽位、block 完成计数、查表映射
- `可靠性信号`：ACK、超时补取、Resend
- `拥塞反馈`：ECN
- `完成条件`：聚合计数满足后返回聚合结果；所有 host 完成后切换到下一 block
